# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Scope

The active code lives in `src/` (Common Lisp). The `C++/` directory is an abandoned earlier attempt ‚Äî ignore it. Design context lives in `notes/Overview.txt` and `notes/Notes.pdf` (the latter is unfiltered, roughly chronological scratch notes).

This is a research/personal project released to the public domain. It began as only the **input component** of a three-part design (Input ‚Üí Processing ‚Üí Output). `Plan.md` (repo root) lays out a phased build toward continual, Hebbian-style learning that adds the Processing and Output components; **Phases 0‚Äì7 are implemented** ‚Äî data-structure foundations, output generation, the association bridge, spreading-activation inference, reward-modulated reinforcement with decay/pruning, an interactive teaching loop (`main`), and persistence. The input‚Üíprocessing‚Üíoutput design is functionally whole and survives restarts: you can teach the running system (input, response) pairs and it reproduces or generalizes them on later inputs, across sessions. **Phase 7** adds a pure-Hebbian **concept graph** (`concepts.lisp`) that generalizes to novel inputs: trained on a starter knowledge base, never-taught "horses"/"birds" are recognized as walking on their legs while legless animals (snakes/fish), vehicles, furniture, and nonsense are excluded. It is auto-populated slot-free from `learn`, persists with the network, and uses an adaptive per-category threshold. `train-from-file` imports a training set (`src/training-set.txt`, 105 facts); `export-kb`/`import-kb` save and reload the whole knowledge base. A broader starter KB (`src/knowledge-base.txt`, ~2,100 facts spanning greetings, a large animal-trait matrix, composition, categories, colors, opposites, arithmetic, **world geography** (countries, capitals, continents, rivers, landmarks), **history** (events, dates, people), space, and simple science) is **auto-learned by `main` on first start** (`*starter-kb*`), so the system answers out of the box. A separate **attention head** (`attention.lisp`), built as Hebbian fast weights, adds copy/binding ‚Äî the original FIRST GOAL `say X ‚Üí X` for a *novel* X (taught only `say dog/cat/house`, it answers `say car ‚Üí car`). On top of these, **conversation memory** resolves follow-ups against the previous turn (after "do dogs have legs?", "and cats?" ‚Üí "do cats have legs?") and **template composition** assembles novel replies from learned fragments (taught "what is a dog/cat ‚Üí a ‚Ä¶ is an animal", it answers "what is a horse?" ‚Üí "a horse is an animal"). **Learned operations** (`operations.lisp`) add *computed* answers over the system's own knowledge ‚Äî taught once, "how many animals do you know" counts them generically and rises as the system learns, and "what is similar to X" returns related concepts ‚Äî and **distributed concept vectors** (`vectors.lisp`, the CMAC / hyperdimensional direction) add non-brittle similarity (dog‚Üîcat clusters emerge from co-occurrence; used for follow-ups and `similar`). See `Plan.md` ¬ß3.5‚Äì¬ß3.8. The original design notes are in `notes/`.

## Running the code

There is no real build system. `src/Makefile` only provides `make clean` to remove Lisp fasl artifacts (`*.fas`, `*.fasl`, `*.lib`, `*.lx64fsl`).

To run, load and start from a Common Lisp REPL **with the working directory set to `src/`**
(so relative paths ‚Äî the training file, the saved knowledge base ‚Äî resolve there). Load the
whole system with one call:

```lisp
(load "load.lisp")   ; loads every component in dependency order; works on all impls
(main)               ; NOT (ai::main) ‚Äî see note below
```

`load.lisp` also defines `(load-system)`, which reloads everything after an edit. Under the
hood it loads each component by pathname; each file `(provide ‚Ä¶)`s, so the internal
`(require ‚Ä¶)` forms are no-ops. (CLISP alone can also `(load "ai.lisp")` directly ‚Äî its
`require` searches the current directory ‚Äî but SBCL/CCL/ECL do not, which is why `load.lisp`
loads by pathname.)

`main` is an interactive teaching loop (Phase 5). Each turn it reads an **input** sentence, prints what it would answer (or "I don't know"), then reads a **teacher** line ‚Äî the correct response, or a confirm word (`yes`/`y`/`right`/`correct`/`ok`) to accept a correct guess ‚Äî and learns from it. It exits on `quit.`/`exit.` (or end-of-input) and prints the input neuron tree. It loads saved memory from `*save-file*` (default `ai-network.kb` in the working directory) on entry and saves it on exit, so learning persists across runs. On a **first** start with no saved memory it learns `*starter-kb*` (`knowledge-base.txt`, ~2,100 facts) so a fresh system already answers questions, copies, and composes out of the box (bind `*starter-kb*` to `nil` to start blank).

**Note on `(main)` vs `(ai::main)`:** `ai.lisp` has no `defpackage`/`in-package` (unlike the component files), so its only function, `main`, is interned into whatever package is current at load time ‚Äî normally `common-lisp-user`. There is no `ai` package; call `(main)`, not `(ai::main)`.

**Input format:** all input is lowercased (`string-downcase` in `getword`), and every sentence must end with a terminator ‚Äî `.`, `!`, or `?` ‚Äî which is read as its own token. This applies to the exit commands too: type `quit.` or `exit.` (a bare `quit` with no terminator just makes the reader block waiting for more input).

**Tests:** a small smoke-test suite lives in `src/tests.lisp` (148 assertions). It loads the components by pathname ‚Äî so it stays loadable on CLISP, SBCL, CCL, and ECL, though `make test` now runs SBCL (Blake's primary) ‚Äî and checks the data-structure foundations, the Output component, the association bridge, inference, the reinforcement/decay dynamics, the interactive teaching loop, persistence (a save‚Üíreset‚Üíreload round-trip), concept-graph generalization (auto-populated, adaptive, persisted), the starter knowledge base (`train-from-file` on `training-set.txt`, then `export-kb`/`import-kb`), the string interface, the attention copy head (`say X ‚Üí X` for novel X), conversation memory (follow-ups), template composition, the broad startup KB (`knowledge-base.txt`), learned operations (count / similar over the knowledge, persisted), distributed concept vectors (similarity, `similar`, persisted), non-brittle membership recognition, and learning from raw text (`read-text`). Tests that touch files bind `*save-file*` / use throwaway paths and delete them. Run it with `make test` (SBCL; non-zero exit on failure) or interactively, from `src/`, with `(load "tests.lisp")`.

For a manual end-to-end check of the teaching loop (lines alternate input / teacher; note the `(main)` line and that input is piped to the REPL, not passed via `clisp -x`, which would hijack `*standard-input*`). This teaches "say hi" ‚Üí "hi there", then re-asks and confirms the correct guess:

```sh
cd src
printf '(load "load.lisp")\n(main)\nsay hi.\nhi there.\nsay hi.\nyes.\nquit.\n(ext:quit)\n' | clisp -q
```

## Architecture

The system converts input sentences into a network of neuron structures, maps them to output structures, and generates responses. Eleven source files (`data-structures`, `line-input`, `input`, `output`, `processing`, `persist`, `concepts`, `attention`, `operations`, `vectors` are each their own package; `ai.lisp` is the package-less top level):

- **`data-structures.lisp`** (package `data-structures`) ‚Äî defines `neuron`, `named-neuron` (neuron + `name` slot for word neurons), and `dendrite` (an edge to another neuron, carrying a `weight`, a `kind` ‚Äî `:sequence` / `:extender` / `:association` ‚Äî and a `from` back-pointer to its source, set on `:association` edges so decay can prune them). Holds the global `*dictionary*` hash table (word string ‚Üí named-neuron), the `*next-neuron-id*` counter, and the `*output-roots*` / `*associations*` registries (empty for now ‚Äî reserved for the Processing/Output components per `Plan.md`). `reset` clears all of them. `dump-dictionary` recursively prints the network.

- **`line-input.lisp`** (package `line-input`) ‚Äî tokenizes stdin into words terminated by `.`, `!`, or `?`. `create-line` returns a list of `named-neuron`s for one sentence, interning new words into `*dictionary*` via `intern-word`. Also exports `intern-words` (intern a list of word strings without reading stdin ‚Äî used by inference) and the `add` macro (push onto a list variable).

- **`input.lisp`** (package `input`) ‚Äî the input algorithm, extracted from `ai.lisp` (Phase 3) so both `main` and `processing` can call it without a circular dependency. `build-structure` (described below) turns a sentence into the neuron network and returns the `active` meaning neurons; `connect`, `connects-to`, and `find-connecting-neuron` are its helpers.

- **`ai.lisp`** ‚Äî the package-less top-level entry point: the interactive teaching-loop `main` (read input ‚Üí `respond` ‚Üí read teacher line ‚Üí `learn`) plus `train-from-file` (bulk-learn a training set of `input => answer` lines), **conversation memory** (`ask` resolves a follow-up against the previous turn ‚Äî `*last-turn*` ‚Äî by replacing its most concept-similar word, e.g. "and cats?" ‚Üí "do cats have legs?", then answers via `infer-answer`; `main` does the same), **learning from raw text** (`read-text` / `read-text-file`: each sentence feeds the distributed-vector co-occurrence for *unsupervised* similarity, and simple declarative patterns ‚Äî "X is the Y of Z" ‚Üí relational, "X is the largest Y" ‚Üí superlative ("what is the largest Y"), "X is a Y" ‚Üí membership, "X was a ROLE" ‚Üí "who was X" ‚Äî also teach facts; questions are skipped. A ~370-sentence corpus `src/prose.txt` (history, world geography, and science) ships with it: `(read-text-file "prose.txt")` teaches capitals, people (by full name and surname), and relational facts ("what is the study of stars" ‚Üí astronomy), and clusters the new entities by similarity), and the `require`/`use-package` forms that pull the whole system together. (The input algorithm now lives in `input.lisp`.)

- **`output.lisp`** (package `output`) ‚Äî the start of the **Output** component (Phase 1 of `Plan.md`), the mirror of the input network. `build-output-structure` builds (or reuses, keyed by text in `*responses*`) a divergent extender chain `root ‚Üí w1 ‚Üí w2 ‚Üí ‚Ä¶` for a response and registers the root in `*output-roots*`; `produce-output` walks the chain and returns the emitted words; `output-sentence` joins them. Output nodes are fresh neurons, never the input word neurons in `*dictionary*`, so the two networks never share an extender chain.

- **`processing.lisp`** (package `processing`) ‚Äî the start of the **Processing** component (Phase 2 of `Plan.md`), the bridge between the two networks. `associate(input-endings, output-root)` wires every input "meaning" neuron (the `active` set returned by `build-structure`) to an output root with a weighted `:association` dendrite ‚Äî created at `*assoc-initial-weight*` or strengthened by `*assoc-strengthen*` (clamped to `*assoc-max-weight*`) and registered in `*associations*`. These cross-component edges are the only ones later phases reinforce and decay; structural `:sequence`/`:extender` edges are permanent. `respond(input-words)` does inference (Phase 3): it builds the input, spreads each meaning neuron's association `weight`s into the output roots' `current-value`, and returns the highest-scoring root's response (winner-take-all above `threshold`) ‚Äî or `nil` ("I don't know") if nothing is activated. `learn(input-words, correct-words)` is one continual-learning turn (Phase 4): it reinforces the correct pathway, weakens a wrong guess, runs a global `decay-associations` step (shrinking unused weights, pruning those below `*assoc-prune-threshold*`, GC'ing orphaned roots), and homeostatically adapts the correct root's threshold ‚Äî the rule `Œîw = Œ∑¬∑pre¬∑post¬∑r ‚àí Œª¬∑w` from `Plan.md` ¬ß3.4.

- **`persist.lisp`** (package `persist`) ‚Äî persistence (Phase 6). `save-network`/`load-network` serialise the whole graph to one readable s-expression (neurons written flat and keyed by `id`, every reference as an id, rebuilt in two passes via an id‚Üíneuron map so shared structure and the cyclic `from` back-pointers survive) and reload it; `main` loads `*save-file*` on entry and saves on exit, so learning persists across restarts. `export-kb`/`import-kb` save and reload the entire knowledge base (input network, outputs, associations, **and** the concept graph) to/from a chosen file.

- **`concepts.lisp`** (package `concepts`) ‚Äî the Phase 7 **concept graph**, where generalization lives. `relate(subject, predicate, answer)` Hebbianly strengthens an edge between a subject concept (a shared `*dictionary*` neuron) and a `(predicate:answer)` state neuron in `*concept-graph*`; `category-strength`/`recognizes-p` answer "does X belong to this category?" by degree-normalized spreading activation. Similar subjects share state neighbours, so a novel word generalizes in while dissimilar ones (and nonsense) are excluded ‚Äî no slots, no rules. Edges live in `*concept-graph*` (not in `axon` slots) so the input network is untouched. The graph is **auto-populated from `learn`** (`note-relationship` relates each input word to its complement-frame + answer; the slot-free `infer-p`/`infer-strength` query tries every word-as-subject decomposition), the membership threshold is **adaptive** (relative to each category's members, via `member-baseline` √ó `*concept-fraction*`), and it **persists** with the rest of the network. `infer-answer` returns the best-supported answer to an input (used by conversational `ask`), and `concept-similarity` (shared-neighbour count) drives follow-up resolution. See `Plan.md` ¬ß3.5 / Phase 7.

- **`attention.lisp`** (package `attention`) ‚Äî a transformer-like **attention head realized as Hebbian fast weights**: each token/position gets a deterministic vector, a sentence is bound as `M = Œ£ role·µ¢ ‚äó token·µ¢` (outer products), and a role-query retrieves a token (`r¬∑M`, then decode) ‚Äî attention with keys = roles, values = token vectors, no backprop. This gives the **copy/binding** capability the associative net and concept graph lack ‚Äî the project's original FIRST GOAL, `say X ‚Üí X` for a *novel* X. `learn` calls `note-copy` to learn (in `*copy-cues*`, persisted) that a cue word triggers "copy the next word"; once a cue passes `*copy-threshold*`, `respond` fires `copy-response` (an induction head) before falling back to associative spreading. `attention-experiment.lisp` is a standalone proof-of-concept. This file also holds **template composition** (`note-template`/`compose`, `*templates*`): `learn` records a multi-word answer that reuses an input word as a `frame ‚Üí template-with-:slot`; the genuine slot recurs across examples so it accumulates strength, and `respond` (after copy, before associative recall) fills a strong template's slot with the input word ‚Äî composing a sentence never seen verbatim (taught "what is a dog/cat", it answers "what is a horse?" ‚Üí "a horse is an animal").

- **`operations.lisp`** (package `operations`) ‚Äî **learned operations over the system's own knowledge** (understanding as a *computed* answer, not a stored fact; the answer is recomputed over current knowledge, so it changes as the system learns). A small set of general primitives ‚Äî `count` (members of a category) and `similar` (nearest concepts) ‚Äî plus a **learned mapping** from a question phrasing to (operation + slot): teaching "how many animals do you know ‚Üí count animals" records a `frame ‚Üí operation` in `*op-templates*` (persisted) that generalizes to any category via the slot. `count`'s membership is found **generically** ‚Äî the positive "are/is _ C" frame the most subjects share (the real category accumulates members; idiosyncratic decompositions don't), no per-category code. `learn` routes an operation-answer to `note-operation`; `respond` runs a matching operation before copy/compose/recall. `recognized-member-p` answers category membership **non-brittlely** ‚Äî framed (crisp) or, for a word never explicitly told, by k-NN resemblance in the vector space (a novel word taught a few traits is recognized as an animal; non-members rejected).

- **`vectors.lisp`** (package `vectors`) ‚Äî **distributed concept vectors** (the CMAC / sparse-distributed-memory / hyperdimensional direction), for non-brittle *similarity*. Each concept is a high-dimensional vector built **online** by superposing the random codes of the words it co-occurs with (Hebbian accumulation of counts in `*cooccur*`, persisted; a few vector adds per fact, no backprop), IDF-weighted and mean-centered so discriminating context decides. `similarity`/`nearest`/`centroid-of` make meaning geometry: similar concepts cluster (dog‚Üîcat, red‚Üîblue) and a novel word generalizes by proximity, with no cliff. Used for conversational follow-up resolution and the `similar` operation. **Head-to-head finding:** vectors win at similarity / graceful generalization; the discrete concept graph stays better for crisp counting (there is no clean boundary in the vector space), so the two are complementary. `concept-vectors-experiment.lisp` (and `procedure-experiment.lisp`) are standalone proofs-of-concept.

### How `build-structure` works

This is the non-obvious part ‚Äî read `notes/Overview.txt` alongside the code. For each word in the input sentence, the system maintains an `active` list of "frontier" neurons representing the running partial sentence and its subset variants. For each new word neuron:

1. The word's **extender neuron** (`neuron-extender`, lazily created via `get-extender`) represents "this word in sequence" and seeds the next frontier.
2. For each prior frontier neuron `pn`, the system also adds (a) `pn`'s extender (representing the prior partial-sentence advancing past the new word ‚Äî the "missing word" combination) and (b) either an existing neuron that both `pn` and the new word already connect to, or a freshly created joining neuron (`connect pn (new-next-neuron neuron)`).

The result: a single sentence builds neurons not only for the full sequence but for every subset preserving order, so the network is less brittle to exact wording. `find-connecting-neuron` is what makes repeated training reuse existing nodes instead of duplicating structure. `build-structure` returns this final `active` frontier ‚Äî the candidate "meaning" neurons that the Processing component consumes (`associate` wires them to an output root during teaching; `respond` fires them during inference).

Each `neuron` has an `axon` (list of outgoing `dendrite`s) and at most one designated `extender` neuron, which is also present in the axon list. `connects-to` and `find-connecting-neuron` walk the axon to detect existing edges ‚Äî and skip `:association` edges, so the cross-component links added by `processing` never interfere with input structure-building.

**Activation slots:** `neuron`'s `current-value` and `threshold` (both `single-float`) and `dendrite`'s `weight` belong to the Processing side, not structure-building. `associate` sets association `weight`s; `respond` spreads them into roots' `current-value` and fires the one whose value exceeds its `threshold`; `learn` homeostatically adapts `threshold` (it drifts toward a small fraction of a root's activation). The input algorithm never touches these. (`*next-neuron-id*` remains debug-only ‚Äî see below.)

## Conventions

- `data-structures`, `line-input`, `input`, `output`, `processing`, and `persist` are each their own package and use `(provide ...)` / `(require ...)` so they can be loaded individually (in dependency order). `ai.lisp` is the top-level entry point: it `require`s and `use-package`s all six but has no `defpackage`/`in-package`/`provide` of its own, so its symbols (`main` and its helpers) land in the current package (see the run note above).
- New exported symbols must be added to the `:export` list of the relevant `defpackage`.
- Package names and exported symbols are written as case-sensitive strings (e.g. `(defpackage "data-structures" ‚Ä¶)`, `"NEURON-AXON"`), so the exact casing matters ‚Äî export strings are uppercase, package names lowercase. Because the package names are lowercase, you can't reach their symbols with the reader's `pkg:sym` syntax (the reader upcases `pkg` to `DATA-STRUCTURES`, which doesn't exist) ‚Äî use string designators (`(use-package "data-structures")`) or `(find-symbol "RESET" "data-structures")`. `ai.lisp` does exactly this to `shadowing-import` `reset` before `use-package`, because SBCL's `CL-USER` inherits `SB-PROFILE:RESET` and would otherwise raise a name conflict.
- `*next-neuron-id*` / a neuron's `id` are not part of the algorithm (only `dump-neuron`'s debug output uses the id for display) ‚Äî but `persist` reuses each neuron's `id` as its serialization key, so ids must stay unique.
