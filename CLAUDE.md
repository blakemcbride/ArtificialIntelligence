# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Scope

The active code lives in `src/` (Common Lisp). Design context lives in `notes/Overview.txt` and `notes/Notes.pdf` (the latter is unfiltered, roughly chronological scratch notes).

This is a research/personal project released to the public domain. It began as only the **input component** of a three-part design (Input → Processing → Output). `Plan.md` (repo root) lays out a phased build toward continual, Hebbian-style learning that adds the Processing and Output components; **Phases 0–10 are implemented** — data-structure foundations, output generation, the association bridge, spreading-activation inference, reward-modulated reinforcement with decay/pruning, an interactive teaching loop (`main`), persistence, **generation** (Phase 8), **learned relation discovery** (Phase 9 — "is a" is learned, not hardcoded), and **in-context learning via an induction head** (Phase 10 — a two-layer attention circuit that continues a prompt's pattern with no training). The input→processing→output design is functionally whole and survives restarts: you can teach the running system (input, response) pairs and it reproduces or generalizes them on later inputs, across sessions. **Phase 7** adds a pure-Hebbian **concept graph** (`concepts.lisp`) that generalizes to novel inputs: trained on a starter knowledge base, never-taught "horses"/"birds" are recognized as walking on their legs while legless animals (snakes/fish), vehicles, furniture, and nonsense are excluded. It is auto-populated slot-free from `learn`, persists with the network, and uses an adaptive per-category threshold. `train-from-file` imports a training set (`src/generalization-test.txt`, 105 facts); `export-kb`/`import-kb` save and reload the whole knowledge base. A broader starter KB (`src/knowledge-base.txt`, ~2,100 facts spanning greetings, a large animal-trait matrix, composition, categories, colors, opposites, arithmetic, **world geography** (countries, capitals, continents, rivers, landmarks), **history** (events, dates, people), space, and simple science) is **auto-learned by `main` on first start** (`*starter-kb*`), so the system answers out of the box. A separate **attention head** (`attention.lisp`), built as Hebbian fast weights, adds copy/binding — the original FIRST GOAL `say X → X` for a *novel* X (taught only `say dog/cat/house`, it answers `say car → car`). On top of these, **conversation memory** resolves follow-ups against the previous turn (after "do dogs have legs?", "and cats?" → "do cats have legs?") and **template composition** assembles novel replies from learned fragments (taught "what is a dog/cat → a … is an animal", it answers "what is a horse?" → "a horse is an animal"). **Learned operations** (`operations.lisp`) add *computed* answers over the system's own knowledge — taught once, "how many animals do you know" counts them generically and rises as the system learns, and "what is similar to X" returns related concepts — and **distributed concept vectors** (`vectors.lisp`, the CMAC / hyperdimensional direction) add non-brittle similarity (dog↔cat clusters emerge from co-occurrence; used for follow-ups and `similar`). **Generation** (`generation.lisp`, Phase 8) finally adds the Output ambition the rest of the system lacked — *assembling a new sentence* rather than retrieving one: "tell me about France" → "France is a country. It is in europe. Its capital is paris…" (a multi-fact paragraph aggregated from `(subject relation object)` triples), and "why is a cat an animal" → "a cat is an animal because a cat is a mammal, and a mammal is an animal" (an is-a chain). Both are grown by online counting (Hebbian, no backprop) and persist. See `Plan.md` §3.5–§3.8 and Phase 8. The original design notes are in `notes/`.

## Running the code

There is no real build system. `src/Makefile` only provides `make clean` to remove Lisp fasl artifacts (`*.fas`, `*.fasl`, `*.lib`, `*.lx64fsl`).

To run, load and start from a Common Lisp REPL **with the working directory set to `src/`**
(so relative paths — the training file, the saved knowledge base — resolve there). Load the
whole system with one call:

```lisp
(load "load.lisp")   ; loads every component in dependency order; works on all impls
(main)               ; NOT (ai::main) — see note below
```

The launcher script **`./sbcl-llm`** (repo root) does all of this: it starts SBCL with a 16 GB
heap (`--dynamic-space-size 16384`), loads `load.lisp`, and enters `main` — the recommended way
to run a large-corpus training session. The training workflow (bulk configuration, very large
files, switching back for everyday use) is documented in **`Training.md`**.

`load.lisp` also defines `(load-system)`, which reloads everything after an edit. Under the
hood it loads each component by pathname; each file `(provide …)`s, so the internal
`(require …)` forms are no-ops. (CLISP alone can also `(load "ai.lisp")` directly — its
`require` searches the current directory — but SBCL/CCL/ECL do not, which is why `load.lisp`
loads by pathname.)

`main` is an interactive teaching loop (Phase 5). Each turn it reads an **input** sentence, prints what it would answer (or "I don't know"), then reads a **teacher** line — the correct response, or a confirm word (`yes`/`y`/`right`/`correct`/`ok`) to accept a correct guess — and learns from it. It exits on `.quit`/`.exit` (or end-of-input). It loads saved memory from `*save-file*` (default `auto-save.kb` in the working directory) on entry and saves it on exit, so learning persists across runs. On a **first** start with no saved memory it learns `*starter-kb*` (`knowledge-base.txt`, ~2,100 facts) so a fresh system already answers questions, copies, and composes out of the box (bind `*starter-kb*` to `nil` to start blank).

**Loop commands** (each typed alone on a line, beginning with a period so they never collide with a sentence): `.help` (the command list), `.stats` (print `system-stats`), `.list [DIR]` (list the `.kb` files in DIR, default the current directory), `.save FILE` (write the knowledge base to FILE), `.load FILE` (clear, then load the knowledge base from FILE), `.read FILE` (learn from FILE via `load-knowledge`, which auto-routes each line: `input => answer` lines → `learn`, other non-comment lines → `read-text` prose, so one file may mix both — and **resumes from where it last stopped**: a per-file byte offset in `*read-offsets*` is advanced and persisted, so with `.set read-max-mb N` repeated `.read FILE` calls walk a huge corpus in N-MB slices, even across sessions), `.rewind FILE` (forget that offset so the next `.read` starts over), `.merge FILE` (sum another `.kb`'s **learned counts** — co-occurrence, transitions, sentence-starts, facts, relations — into the current model without resetting; combines shards from parallel/multi-process bulk reads, via `persist:merge-kb`; the neuron/concept graph is not merged, only the additive stores), `.config` (show the tunable model-cap parameters and current model sizes), `.set NAME VALUE` (change a tunable live, e.g. `.set max-cooccur 200000`, `.set read-extract off`; `off`/`nil` = unlimited), and `.quit`/`.exit` (save and exit). Command lines are read raw, so a filename keeps its case and embedded dots (`.save MyKB.kb`); `parse-command`/`run-command` in `ai.lisp` implement them. **Model caps** (`*max-vocab*` / `*max-cooccur*` / `*max-transitions*` / `*max-facts*`, plus `*read-max-mb*` and `*read-extract*`, all in `ai.lisp`, exposed via `*tunables*`): default NIL = unlimited; when set, over-cap stores are pruned to their strongest entries every `*prune-every*` learned sentences (`enforce-caps`), so an arbitrarily large corpus can be ingested in fixed memory. `*read-extract*` off skips the heavy supervised path (build-structure / concept graph) for cheap bulk reading. **Per-learner toggles** (`*read-cooccur*` / `*read-relations*` / `*read-facts-p*` / `*read-transitions-p*`, exposed as `.set read-cooccur off` etc., all default on) let the lean bulk path drop individual statistical learners — chiefly the O(words²) co-occurrence learner — for huge/noisy corpora. **Parallel bulk reading** (`*read-workers*`, `.set read-workers N`; SBCL only, default 1 = sequential): when `read-extract` is off and N > 1, `read-text-file` / `load-knowledge` fan the per-sentence learning out across N worker threads by **shard-and-merge** — one reader thread streams + tokenizes (the cheap I/O-bound part) and a backpressure-bounded mailbox feeds batches to workers that each learn into *private* count stores (dynamic rebindings, zero contention), summed at the end via `persist:merge-worker-stores`; because the learners are additive counters the merged result is **exact** for the pure counters (the order-dependent relation-discovery facts differ marginally). Supervised `=>` pairs in a parallel `load-knowledge` are deferred and applied sequentially after the parallel pass (they mutate the shared graph). It falls back to the sequential reader on non-thread builds, so the code stays portable. The current tunable values are **saved in the `.kb` and restored on load** (`.save` records the settings in effect at save time; `.load`/`import-kb` re-applies them, via the `persist:*tunable-provider*` / `persist:*tunable-restorer*` hooks `ai.lisp` installs), so model caps and `read-extract` persist with the knowledge base and across sessions; an older `.kb` with no saved tunables leaves them unchanged. `.save`/`.load` also **set the active file** — they `(setf *save-file* FILE)` — so the loaded/saved file becomes what `main` auto-saves to on exit (and auto-loads on the next `(main)` in the same session); the default active file is `auto-save.kb`.

**Note on `(main)` vs `(ai::main)`:** `ai.lisp` has no `defpackage`/`in-package` (unlike the component files), so its only function, `main`, is interned into whatever package is current at load time — normally `common-lisp-user`. There is no `ai` package; call `(main)`, not `(ai::main)`.

**Input format:** all input is lowercased (`string-downcase`). A sentence terminator is `.`, `!`, or `?`, read as its own token — but a period only ends a sentence at end-of-line or when followed by a space (`terminator-p` in `line-input.lisp`), so an embedded period stays inside its word (a filename like `file.kb`, a decimal like `3.14`). The interactive loop reads each prompt as a whole raw line, so in the loop a trailing terminator is optional. (Note `create-line`, the stdin token reader still used elsewhere, does require a terminator to know a sentence has ended.) Loop commands begin with a leading period — `.quit`, `.stats`, `.save FILE`, etc. — see **Loop commands** above; a bare `quit`/`exit` word also still stops the loop.

**Tests:** a small smoke-test suite lives in `src/tests.lisp` (204 assertions on SBCL; a couple are SBCL-thread-only). It loads the components by pathname — so it stays loadable on CLISP, SBCL, CCL, and ECL, though `make test` now runs SBCL (Blake's primary) — and checks the data-structure foundations, the Output component, the association bridge, inference, the reinforcement/decay dynamics, the interactive teaching loop, persistence (a save→reset→reload round-trip), concept-graph generalization (auto-populated, adaptive, persisted), the starter knowledge base (`train-from-file` on `generalization-test.txt`, then `export-kb`/`import-kb`), the string interface, the attention copy head (`say X → X` for novel X), conversation memory (follow-ups), template composition, the broad startup KB (`knowledge-base.txt`), learned operations (count / similar over the knowledge, persisted), distributed concept vectors (similarity, `similar`, persisted), non-brittle membership recognition, learning from raw text (`read-text`), the facts counter / `system-stats`, the contextual sentence terminator (embedded periods kept), the leading-period loop commands (`.save`/`.load`/`.list`, including that `.load` makes the loaded file the auto-save target), and **generation** (describe / why, QA- and prose-derived facts, the transition model, persistence). Tests that touch files bind `*save-file*` / use throwaway paths and delete them. Run it with `make test` (SBCL; non-zero exit on failure) or interactively, from `src/`, with `(load "tests.lisp")`.

For a manual end-to-end check of the teaching loop (lines alternate input / teacher; note the `(main)` line and that input is piped to the REPL, not passed via `clisp -x`, which would hijack `*standard-input*`). This teaches "say hi" → "hi there", then re-asks and confirms the correct guess:

```sh
cd src
printf '(load "load.lisp")\n(main)\nsay hi.\nhi there.\nsay hi.\nyes.\n.quit\n(ext:quit)\n' | clisp -q
```

## Architecture

The system converts input sentences into a network of neuron structures, maps them to output structures, and generates responses. Fourteen source files (`data-structures`, `line-input`, `input`, `output`, `processing`, `persist`, `concepts`, `attention`, `operations`, `vectors`, `generation`, `relations`, `induction` are each their own package; `ai.lisp` is the package-less top level):

- **`data-structures.lisp`** (package `data-structures`) — defines `neuron`, `named-neuron` (neuron + `name` slot for word neurons), and `dendrite` (an edge to another neuron, carrying a `weight`, a `kind` — `:sequence` / `:extender` / `:association` — and a `from` back-pointer to its source, set on `:association` edges so decay can prune them). Holds the global `*dictionary*` hash table (word string → named-neuron), the `*next-neuron-id*` counter, and the `*output-roots*` / `*associations*` registries (empty for now — reserved for the Processing/Output components per `Plan.md`). `reset` clears all of them. `dump-dictionary` recursively prints the network.

- **`line-input.lisp`** (package `line-input`) — tokenizes input into words; a sentence ends at `!`, `?`, or a period that is at end-of-line or followed by a space (`terminator-p`), so embedded periods stay inside a word (`file.kb`, `3.14`). `create-line` returns a list of `named-neuron`s for one sentence read from stdin, interning new words into `*dictionary*` via `intern-word`; `tokenize` does the same splitting for an in-memory string (used by `as-words` and the interactive loop). Also exports `intern-words` (intern a list of word strings without reading stdin — used by inference) and the `add` macro (push onto a list variable).

- **`input.lisp`** (package `input`) — the input algorithm, extracted from `ai.lisp` (Phase 3) so both `main` and `processing` can call it without a circular dependency. `build-structure` (described below) turns a sentence into the neuron network and returns the `active` meaning neurons; `connect`, `connects-to`, and `find-connecting-neuron` are its helpers.

- **`ai.lisp`** — the package-less top-level entry point: the interactive teaching-loop `main` (read input → `respond` → read teacher line → `learn`) plus `train-from-file` (bulk-learn a training set of `input => answer` lines), **conversation memory** (`ask` resolves a follow-up against the previous turn — `*last-turn*` — by replacing its most concept-similar word, e.g. "and cats?" → "do cats have legs?", then answers via `infer-answer`; `main` does the same), **learning from raw text** (`read-text` / `read-text-file`: each sentence feeds the distributed-vector co-occurrence for *unsupervised* similarity, and simple declarative patterns — "X is the Y of Z" → relational, "X is the largest Y" → superlative ("what is the largest Y"), "X is a Y" → membership, "X was a ROLE" → "who was X" — also teach facts; questions are skipped. A ~690-sentence corpus `src/prose.txt` (history, world geography, and science) ships with it: `(read-text-file "prose.txt")` teaches capitals, people (by full name and surname), and relational facts ("what is the study of stars" → astronomy), and clusters the new entities by similarity), a **unified loader** (`load-knowledge`: one front door over both modes — it **streams a file in bounded-memory chunks** (`stream-chunks`; never loaded whole, UTF-8-safe — fine for 100GB+ corpora) and routes each line, `input => answer` → `learn`, any other non-comment line → `read-text` prose, so a single file may mix supervised pairs and prose; `train-from-file` / `read-text` underneath; `.read` uses it; bind **`*read-max-mb*`** to cap a huge ingest — note the *file* streams in constant memory but the learned model still grows with what is read), and the `require`/`use-package` forms that pull the whole system together. (The input algorithm now lives in `input.lisp`.)

- **`output.lisp`** (package `output`) — the start of the **Output** component (Phase 1 of `Plan.md`), the mirror of the input network. `build-output-structure` builds (or reuses, keyed by text in `*responses*`) a divergent extender chain `root → w1 → w2 → …` for a response and registers the root in `*output-roots*`; `produce-output` walks the chain and returns the emitted words; `output-sentence` joins them. Output nodes are fresh neurons, never the input word neurons in `*dictionary*`, so the two networks never share an extender chain.

- **`processing.lisp`** (package `processing`) — the start of the **Processing** component (Phase 2 of `Plan.md`), the bridge between the two networks. `associate(input-endings, output-root)` wires every input "meaning" neuron (the `active` set returned by `build-structure`) to an output root with a weighted `:association` dendrite — created at `*assoc-initial-weight*` or strengthened by `*assoc-strengthen*` (clamped to `*assoc-max-weight*`) and registered in `*associations*`. These cross-component edges are the only ones later phases reinforce and decay; structural `:sequence`/`:extender` edges are permanent. `respond(input-words)` does inference (Phase 3): it builds the input, spreads each meaning neuron's association `weight`s into the output roots' `current-value`, and returns the highest-scoring root's response (winner-take-all above `threshold`) — or `nil` ("I don't know") if nothing is activated. `learn(input-words, correct-words)` is one continual-learning turn (Phase 4): it reinforces the correct pathway, weakens a wrong guess, runs a global `decay-associations` step (shrinking unused weights, pruning those below `*assoc-prune-threshold*`, GC'ing orphaned roots), and homeostatically adapts the correct root's threshold — the rule `Δw = η·pre·post·r − λ·w` from `Plan.md` §3.4.

- **`persist.lisp`** (package `persist`) — persistence (Phase 6). `save-network`/`load-network` serialise the whole graph to one readable s-expression (neurons written flat and keyed by `id`, every reference as an id, rebuilt in two passes via an id→neuron map so shared structure and the cyclic `from` back-pointers survive) and reload it; `main` loads `*save-file*` on entry and saves on exit, so learning persists across restarts. `export-kb`/`import-kb` save and reload the entire knowledge base (input network, outputs, associations, the concept graph, the Phase 8 generation stores — facts, transitions, sentence-starts — the Phase 9 learned-relation stores, the `.read` resume offsets (`*read-offsets*`), **and** the `.set` tunables (model caps, `read-extract`, …, via the `*tunable-provider*`/`*tunable-restorer*` hooks)) to/from a chosen file. A cumulative **`*facts-learned*`** counter (bumped by every `learn`, from any source — user, file, prose) is saved/restored too. **`merge-kb`** sums another `.kb`'s additive count stores into the current model (for combining bulk-read shards); **`merge-worker-stores`** does the same from a parallel worker's in-memory tables. The merge helpers (`merge-flat!` / `merge-nested!` / `merge-rel-links!`) exploit that every bulk learner is a counter, so disjoint shards combine exactly by summation (the neuron/concept graph is deliberately not merged). **`system-stats`** prints and returns the system's size: facts learned, vocabulary, neurons, dendrites, output roots, associations, responses, concept states/edges, copy cues, templates, operations, distributed-vector vocabulary, generation facts, transition heads, and learned relation connectors.

- **`concepts.lisp`** (package `concepts`) — the Phase 7 **concept graph**, where generalization lives. `relate(subject, predicate, answer)` Hebbianly strengthens an edge between a subject concept (a shared `*dictionary*` neuron) and a `(predicate:answer)` state neuron in `*concept-graph*`; `category-strength`/`recognizes-p` answer "does X belong to this category?" by degree-normalized spreading activation. Similar subjects share state neighbours, so a novel word generalizes in while dissimilar ones (and nonsense) are excluded — no slots, no rules. Edges live in `*concept-graph*` (not in `axon` slots) so the input network is untouched. The graph is **auto-populated from `learn`** (`note-relationship` relates each input word to its complement-frame + answer; the slot-free `infer-p`/`infer-strength` query tries every word-as-subject decomposition), the membership threshold is **adaptive** (relative to each category's members, via `member-baseline` × `*concept-fraction*`), and it **persists** with the rest of the network. `infer-answer` returns the best-supported answer to an input (used by conversational `ask`), and `concept-similarity` (shared-neighbour count) drives follow-up resolution. See `Plan.md` §3.5 / Phase 7.

- **`attention.lisp`** (package `attention`) — a transformer-like **attention head realized as Hebbian fast weights**: each token/position gets a deterministic vector, a sentence is bound as `M = Σ roleᵢ ⊗ tokenᵢ` (outer products), and a role-query retrieves a token (`r·M`, then decode) — attention with keys = roles, values = token vectors, no backprop. This gives the **copy/binding** capability the associative net and concept graph lack — the project's original FIRST GOAL, `say X → X` for a *novel* X. `learn` calls `note-copy` to learn (in `*copy-cues*`, persisted) that a cue word triggers "copy the next word"; once a cue passes `*copy-threshold*`, `respond` fires `copy-response` (an induction head) before falling back to associative spreading. `attention-experiment.lisp` is a standalone proof-of-concept. This file also holds **template composition** (`note-template`/`compose`, `*templates*`): `learn` records a multi-word answer that reuses an input word as a `frame → template-with-:slot`; the genuine slot recurs across examples so it accumulates strength, and `respond` (after copy, before associative recall) fills a strong template's slot with the input word — composing a sentence never seen verbatim (taught "what is a dog/cat", it answers "what is a horse?" → "a horse is an animal").

- **`operations.lisp`** (package `operations`) — **learned operations over the system's own knowledge** (understanding as a *computed* answer, not a stored fact; the answer is recomputed over current knowledge, so it changes as the system learns). A small set of general primitives — `count` (members of a category) and `similar` (nearest concepts) — plus a **learned mapping** from a question phrasing to (operation + slot): teaching "how many animals do you know → count animals" records a `frame → operation` in `*op-templates*` (persisted) that generalizes to any category via the slot. `count`'s membership is found **generically** — the positive "are/is _ C" frame the most subjects share (the real category accumulates members; idiosyncratic decompositions don't), no per-category code. `learn` routes an operation-answer to `note-operation`; `respond` runs a matching operation before copy/compose/recall. `recognized-member-p` answers category membership **non-brittlely** — framed (crisp) or, for a word never explicitly told, by k-NN resemblance in the vector space (a novel word taught a few traits is recognized as an animal; non-members rejected).

- **`vectors.lisp`** (package `vectors`) — **distributed concept vectors** (the CMAC / sparse-distributed-memory / hyperdimensional direction), for non-brittle *similarity*. Each concept is a high-dimensional vector built **online** by superposing the random codes of the words it co-occurs with (Hebbian accumulation of counts in `*cooccur*`, persisted; a few vector adds per fact, no backprop), IDF-weighted and mean-centered so discriminating context decides. `similarity`/`nearest`/`centroid-of` make meaning geometry: similar concepts cluster (dog↔cat, red↔blue) and a novel word generalizes by proximity, with no cliff. Used for conversational follow-up resolution and the `similar` operation. **Head-to-head finding:** vectors win at similarity / graceful generalization; the discrete concept graph stays better for crisp counting (there is no clean boundary in the vector space), so the two are complementary. `concept-vectors-experiment.lisp` (and `procedure-experiment.lisp`) are standalone proofs-of-concept.

- **`generation.lisp`** (package `generation`) — the **Output component, for real** (Phase 8): *assembling a new sentence* instead of retrieving one. Two pieces, both online counting (Hebbian, no backprop, continual): **content selection** — declarative `(subject relation object)` triples in `*facts*`, captured by `note-facts` (prose via `read-text`) and `note-fact-from-qa` (high-value question frames via `learn`, e.g. `what is the R of S` → `(S R A)`); **surface realization** — render each fact through a per-relation frame and **aggregate several into one paragraph** (`describe-words`, "tell me about X"), plus `why` (an is-a-chain explanation) and a learned next-word model (`*transitions*` / `*sentence-starts*`) backing the lower-fidelity `generate-chain`. `respond` checks `generation-request-p` first and, for a recognized "tell me about / describe / what do you know about / why" request, **owns** it — returns the answer or NIL ("I don't know"), never falling through to brittle recall. Grounding the object in a real fact keeps it from hallucinating. The stores live in `data-structures` (so `reset`/persist stay uniform) and persist with the network. `generation-experiment.lisp` is the standalone proof of concept (and a head-to-head vs. a bare bigram chain).

- **`relations.lisp`** (package `relations`) — **learned relation discovery** (Phase 9), the un-hardcoding of "is a". Instead of the rule-based markers in `extract-fact`/`note-facts`, it *learns* — by Hebbian counting, no backprop, continual — (a) **connectors**: a membership connector ("is a") links many subjects to few category hubs (high subjects-per-category), vs. a relational/verb connector (~one-to-one); (b) **function words**: frequent tokens that are rarely a head (so "animal", a frequent *category*, isn't mistaken for glue); (c) **heads**: the real subject/category in a span is the word that most often serves as a head, so modifiers and relative-clause words fall away. `observe` learns online (short sentences seed positionally; longer ones extract with the current model — online bootstrapping, no stored corpus); `relation-of` parses a sentence into `(subject connector category class)`; `membership-connectors` lists the discovered "is a"-like connectors. It runs **alongside** the hardcoded patterns (the cold-start floor) — `read-text` calls `observe` then emits learned membership facts into `*facts*` — and **takes over with data**: on `prose.txt` it discovers `is a`/`was a`/`is an`/`was an` as membership. Stores (`*rel-links*`, `*rel-head*`, `*rel-freq*`, `*rel-sentences*`) live in `data-structures` and persist. `relation-discovery-experiment.lisp` is the standalone PoC (with iterative bootstrapping).

- **`induction.lisp`** (package `induction`) — **in-context learning via an induction head** (Phase 10): the LLM trick of continuing a prompt's pattern with no training. A two-layer attention circuit built from Hebbian fast weights (outer-product associative memories, no backprop): **layer 1** (previous-token head, positional attention) copies each position's predecessor; a **cleanup nonlinearity** (decode → re-encode) sits between; **layer 2** (induction head, content attention) is keyed by predecessor and returns the token that *followed* the query token's earlier occurrence ("… A B … A" → "B"). Tokens/positions are random codes; the circuit is built **fresh from the prompt each call** — purely in-context, so nothing persists. `complete` predicts the next token, `continue-sequence` extends autoregressively, `induction-request-p`/`respond-induction` let `respond` **own** a `continue <sequence>` request. Depth is essential (layer 2's keys are layer 1's outputs). `attention-stack-experiment.lisp` (multi-hop stacking) and `induction-head-experiment.lisp` (in-context learning) are the standalone PoCs. The layer *roles* are hand-wired here — *learning* the per-layer weights with a local rule is the open follow-on, pursued in the experiment arc documented in `notes/TransformerFromHebbianParts.md` (`learned-attention`/`learned-induction`/`learned-qk-attention`/`deep-composition`/`forward-forward`/`ff-attention`, **`predictive-coding-experiment.lisp`** — predictive coding, validated to reproduce the backprop gradient through depth with a purely local rule; and **`pc-attention-experiment.lisp`** — PC carried into the 2-layer attention induction circuit, a clarifying negative result (PC equals backprop's credit through the attention depth, but joint training from scratch plateaus at chance for *both*, so the binding constraint is joint optimization, not locality)).

### How `build-structure` works

This is the non-obvious part — read `notes/Overview.txt` alongside the code. For each word in the input sentence, the system maintains an `active` list of "frontier" neurons representing the running partial sentence and its subset variants. For each new word neuron:

1. The word's **extender neuron** (`neuron-extender`, lazily created via `get-extender`) represents "this word in sequence" and seeds the next frontier.
2. For each prior frontier neuron `pn`, the system also adds (a) `pn`'s extender (representing the prior partial-sentence advancing past the new word — the "missing word" combination) and (b) either an existing neuron that both `pn` and the new word already connect to, or a freshly created joining neuron (`connect pn (new-next-neuron neuron)`).

The result: a single sentence builds neurons not only for the full sequence but for every subset preserving order, so the network is less brittle to exact wording. `find-connecting-neuron` is what makes repeated training reuse existing nodes instead of duplicating structure. `build-structure` returns this final `active` frontier — the candidate "meaning" neurons that the Processing component consumes (`associate` wires them to an output root during teaching; `respond` fires them during inference).

Each `neuron` has an `axon` (list of outgoing `dendrite`s) and at most one designated `extender` neuron, which is also present in the axon list. `connects-to` and `find-connecting-neuron` walk the axon to detect existing edges — and skip `:association` edges, so the cross-component links added by `processing` never interfere with input structure-building.

**Activation slots:** `neuron`'s `current-value` and `threshold` (both `single-float`) and `dendrite`'s `weight` belong to the Processing side, not structure-building. `associate` sets association `weight`s; `respond` spreads them into roots' `current-value` and fires the one whose value exceeds its `threshold`; `learn` homeostatically adapts `threshold` (it drifts toward a small fraction of a root's activation). The input algorithm never touches these. (`*next-neuron-id*` remains debug-only — see below.)

## Conventions

- `data-structures`, `line-input`, `input`, `output`, `processing`, and `persist` are each their own package and use `(provide ...)` / `(require ...)` so they can be loaded individually (in dependency order). `ai.lisp` is the top-level entry point: it `require`s and `use-package`s all six but has no `defpackage`/`in-package`/`provide` of its own, so its symbols (`main` and its helpers) land in the current package (see the run note above).
- New exported symbols must be added to the `:export` list of the relevant `defpackage`.
- Package names and exported symbols are written as case-sensitive strings (e.g. `(defpackage "data-structures" …)`, `"NEURON-AXON"`), so the exact casing matters — export strings are uppercase, package names lowercase. Because the package names are lowercase, you can't reach their symbols with the reader's `pkg:sym` syntax (the reader upcases `pkg` to `DATA-STRUCTURES`, which doesn't exist) — use string designators (`(use-package "data-structures")`) or `(find-symbol "RESET" "data-structures")`. `ai.lisp` does exactly this to `shadowing-import` `reset` before `use-package`, because SBCL's `CL-USER` inherits `SB-PROFILE:RESET` and would otherwise raise a name conflict.
- `*next-neuron-id*` / a neuron's `id` are not part of the algorithm (only `dump-neuron`'s debug output uses the id for display) — but `persist` reuses each neuron's `id` as its serialization key, so ids must stay unique.
